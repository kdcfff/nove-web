# 用户画像驱动竞品研报前端工作台

## 原始需求

现在要实现对自己的平台上传用户画像，同时要基于对我们的用户画像生成研报。当竞品发生变化时，需要结合我们的用户画像生成研报。需求需要覆盖前后端，用户画像支持更新，可以选择一个竞品进行手动生成研报。研报用的组件库要好看一点，排版美观一点。加功能，不要动原来的代码。

## 初始理解

- 在 Nova 竞品情报工作台中新增“我方平台用户画像”入口。
- 用户可以上传或维护自己平台的用户画像，并支持更新。
- 用户可以选择一个竞品，手动触发基于当前用户画像的研报生成。
- 研报页面需要比现有普通详情更美观，使用更适合报告阅读的组件、版式和信息层级。
- 前端需要覆盖用户画像管理、竞品选择、手动生成、生成状态、研报详情和历史记录。
- 不修改原来的代码的含义为：优先新增页面区块、组件、API 类型和入口；不能破坏现有竞品列表、详情抽屉、目标采集、任务记录和既有报告查看流程。

## 初始交付范围

- 新增用户画像上传/编辑/更新 UI。
- 在竞品情报工作台中提供选择单个竞品生成研报的操作。
- 展示研报生成中、成功、失败和空状态。
- 研报详情使用更美观的组件库或现有 UI 组件组合，重点优化排版、证据区、洞察摘要、用户画像匹配分析、行动建议和来源信息。
- 提供研报历史记录和查看入口。
- 前端 API 类型需要与后端接口契约对齐。

## 初始验收口径

- AC-UPR-FE-001: 用户能进入用户画像管理入口并上传或编辑画像。
- AC-UPR-FE-002: 用户画像保存后能看到更新时间和当前生效状态。
- AC-UPR-FE-003: 用户能选择一个竞品并手动触发研报生成。
- AC-UPR-FE-004: 研报生成过程有明确 loading、success、error 状态。
- AC-UPR-FE-005: 研报详情排版美观，包含洞察摘要、证据、影响到的用户画像、建议动作和生成元信息。
- AC-UPR-FE-006: 新功能不破坏现有竞品情报工作台核心路径。

## 待澄清问题

- 用户画像上传格式是文本编辑、Markdown、表单字段、文件上传，还是多种方式都支持。
- 研报详情使用现有 Element Plus 组合即可，还是引入新的报告渲染组件库。
- 研报是否需要导出、打印或分享。
- 竞品变化后的研报是否需要自动提示或只支持手动生成。

## 澄清记录

### 2026-07-09 研报触发范围

- 本轮结论（可直接引用到 solution）：MVP 采用“手动为主，变化后提示可生成”的触发口径。
- 本轮新增/更新的约束：
  - 竞品发生有效变化后，不自动生成画像研报。
  - 前端应在竞品详情、任务/报告上下文或变化提示区提供“可生成画像研报”的入口。
  - 用户需要先选择单个竞品，再基于当前有效用户画像手动触发研报生成。
  - 前端可以展示变化后的生成提示或入口，但首版不承诺自动生成与自动通知。
- 关键决策：研报触发范围 → 选择“手动为主，变化后提示可生成”。

### 2026-07-09 用户画像录入方式

- 本轮结论（可直接引用到 solution）：MVP 用户画像采用“结构化表单 + Markdown 补充说明”的录入和更新方式。
- 本轮新增/更新的约束：
  - 前端提供结构化表单编辑核心画像信息。
  - 前端提供 Markdown 补充说明编辑区，用于承载自由描述、用户研究摘录、业务背景和非结构化洞察。
  - 保存后需要展示当前生效状态和更新时间。
  - 文件上传与自动解析不纳入首版交付范围。
- 关键决策：用户画像录入方式 → 选择“结构化表单 + Markdown 补充说明”。

### 2026-07-09 用户画像版本策略

- 本轮结论（可直接引用到 solution）：MVP 保留用户画像版本快照，但不提供回滚 UI。
- 本轮新增/更新的约束：
  - 每次保存用户画像应形成一个可追溯版本。
  - 研报生成记录需要展示生成时使用的用户画像版本或版本时间。
  - 前端首版只展示当前生效画像、更新时间和研报使用的画像版本信息。
  - 首版不提供版本回滚、版本差异对比或恢复旧版能力。
- 关键决策：用户画像更新策略 → 选择“保留版本快照，但不做回滚 UI”。

### 2026-07-09 研报详情展示形态

- 本轮结论（可直接引用到 solution）：研报详情首版引入新的报告/文档渲染组件库，而不是仅使用现有 Element Plus 组合或整篇 Markdown 渲染。
- 本轮新增/更新的约束：
  - 前端需要选型一个适合报告阅读体验的组件库或文档渲染组件，用于提升研报排版和信息层级。
  - 研报详情应支持摘要、关键洞察、用户画像影响、证据卡片、建议动作和生成元信息的美观展示。
  - 组件库选型需要兼容当前 Vue 前端技术栈、构建流程和主题风格。
- 关键决策：研报详情展示形态 → 选择“引入新的报告/文档渲染组件库”。

### 2026-07-09 研报组件库选型口径

- 本轮结论（可直接引用到 solution）：新组件库选型口径为 Markdown 报告组件库。
- 本轮新增/更新的约束：
  - 前端应引入 Vue 生态可用的 Markdown 报告阅读/预览组件，而不是富文本块编辑器或图表故事组件。
  - 研报详情需要利用 Markdown 组件提升目录、表格、引用、代码块、分段层级和长文阅读体验。
  - 前端仍需围绕 Markdown 正文补充摘要、证据、画像影响、建议动作和元信息等报告型布局。
- 关键决策：新组件库选型口径 → 选择“Markdown 报告组件库”。

### 2026-07-09 研报导出与分享范围

- 本轮结论（可直接引用到 solution）：画像研报首版只支持在线查看，不做导出、打印优化或分享链接。
- 本轮新增/更新的约束：
  - 前端不需要在 MVP 中提供 PDF、Word、打印优化或分享链接入口。
  - 研报详情和历史记录需要支持在线查看。
  - 报告组件库选型优先服务在线阅读体验，而不是导出版式还原。
- 关键决策：研报导出与分享范围 → 选择“首版只支持在线查看，不做导出/分享”。

### 2026-07-09 画像研报 AI 生成链路

- 本轮结论（可直接引用到 solution）：画像研报生成优先复用现有竞品情报 AI 分析/报告生成链路。
- 本轮新增/更新的约束：
  - 前端把“用户画像版本 + 单个竞品 + 触发来源”作为生成请求的核心上下文。
  - 研报详情展示结构化字段和 Markdown 正文，而不是跳转到聊天运行时作为主体验。
  - 不把聊天/AgentScope 交互作为画像研报首版的必需入口。
- 关键决策：画像研报生成链路 → 选择“复用现有竞品情报 AI 分析/报告生成链路”。

### 2026-07-09 用户画像结构化字段

- 本轮结论（可直接引用到 solution）：用户画像结构化表单首版采用轻量画像字段。
- 本轮新增/更新的约束：
  - 结构化字段包含：目标用户、核心场景、关键痛点、购买/使用动机、决策因素、反对理由、产品定位、重点关注指标。
  - 前端需要提供清晰的分组表单和 Markdown 补充说明编辑区。
  - 首版不扩展到完整市场研究字段，如行业、地区、生命周期、渠道、价格敏感度和细分人群矩阵。
- 关键决策：用户画像结构化字段 → 选择“轻量画像字段”。
