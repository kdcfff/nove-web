# 用户画像驱动竞品研报前端工作台方案决策

## 0. 基本信息

- 需求标识（分支 / ID）：`003-user-profile-research-report`
- 作者 / 参与评审：Codex / Nova
- 状态：draft
- 最后更新：2026-07-09
- 关联链接：`requirements/raw.md`

## 1. 结论摘要

- 一句话目标：在竞品情报工作台中新增我方用户画像管理和画像研报在线阅读体验，让用户能选择单个竞品手动生成结合画像的研报。
- In / Out 边界：MVP 包含画像编辑、版本信息展示、竞品手动生成、生成状态、研报历史和 Markdown 报告详情；不包含自动生成、文件上传解析、回滚、导出、分享。
- 推荐方案：在 `/intelligence` 工作台内增量扩展画像管理与画像研报体验，引入 Markdown 报告阅读组件，并保留现有竞品列表、详情、目标、任务和普通报告路径。
- 优先验证点：`V-FE-001` Markdown 组件兼容当前 Vue/Vite 技术栈；`V-FE-002` 报告详情版式能表达结构化字段与 Markdown；`V-FE-003` 新入口不破坏现有工作台核心路径。

## 2. 推荐方案

- 方案名：工作台内聚的 Markdown 画像研报方案
- 主流程 / 关键机制：
  1. 在 `/intelligence` 工作台新增我方用户画像入口，提供轻量结构化表单和 Markdown 补充说明编辑区。
  2. 保存画像后展示当前生效状态、更新时间和版本信息。
  3. 在竞品详情、变化/报告上下文或操作区提供“生成画像研报”入口，用户选择单个竞品手动触发。
  4. 触发后展示生成中、成功、失败状态，并支持进入画像研报历史。
  5. 研报详情使用 Markdown 报告阅读组件承载正文，同时用工作台自定义布局展示摘要、关键洞察、用户画像影响、证据卡片、建议动作和元信息。
  6. 首版在线查看，不提供导出、打印优化和分享链接。
- 关键边界/取舍：
  - 新能力在现有 `/intelligence` 内增量扩展，不重建工作台导航结构。
  - 画像首版使用结构化表单 + Markdown 补充说明，不做文件上传和自动解析。
  - 保留版本快照的展示信息，但不提供回滚、差异对比或恢复旧版 UI。
  - 研报展示引入 Markdown 报告组件库，但仍需要围绕报告阅读设计清晰的信息层级。
  - 不把聊天/AgentScope 页面作为画像研报主体验入口。
- 为什么选它：
  - `raw.md` 已确认用户要“选择一个竞品进行手动生成研报”，且希望研报组件库好看、排版美观。
  - 项目知识库显示 `/intelligence` 是竞品情报工作台入口，`src/api/intelligence` 是前端契约入口。
  - 当前前端技术栈是 Vue 3、Vite、Element Plus、Pinia、UnoCSS，适合增量引入 Vue Markdown 报告阅读组件。

## 3. 备选方案

### 3.1 备选方案：仅用现有 Element Plus 组合

- 核心机制：不新增组件库，用 Element Plus 卡片、描述列表、标签、时间线和现有 Markdown 样式拼出研报详情。
- 主流程：
  1. 画像入口和生成入口照常新增。
  2. 研报详情用 Element Plus 组件展示结构化字段。
  3. Markdown 正文使用现有样式或 `pre` 展示。
  4. 不新增前端依赖。
- 边界与取舍：
  - 依赖风险最低，但长文报告阅读体验和目录/表格/引用支持较弱。
  - 视觉提升可能达不到“组件库好看一点”的诉求。
- 适用前提：
  - 项目禁止新增依赖。
  - 研报正文很短，主要靠结构化卡片表达。
- 不选原因：用户已明确选择引入新的 Markdown 报告组件库。

### 3.2 备选方案：富文本/块编辑器方案

- 核心机制：引入 Notion-like 或富文本块编辑组件，以块结构展示并编辑研报。
- 主流程：
  1. 后端输出块结构协议。
  2. 前端渲染块编辑器。
  3. 用户可编辑或重排研报块。
  4. 保存编辑后的报告结构。
- 边界与取舍：
  - 表达能力强，但需要编辑器状态、序列化协议、权限和样式适配。
  - 会把“在线查看研报”扩展成“研报编辑系统”。
- 适用前提：
  - 用户需要二次编辑、协作和发布研报。
  - 后端已支持块结构文档存储。
- 不选原因：MVP 只要求在线查看，富文本编辑会显著抬高实现复杂度。

### 3.3 备选方案：独立研报中心页面

- 核心机制：不在 `/intelligence` 内扩展，而是新增独立研报中心，统一管理所有画像研报。
- 主流程：
  1. 用户进入研报中心。
  2. 从中心选择竞品生成研报。
  3. 中心展示历史和详情。
  4. 竞品工作台只提供跳转。
- 边界与取舍：
  - 信息架构清晰，但会分散竞品上下文。
  - 需要新增路由、导航和权限口径。
- 适用前提：
  - 研报未来跨多个业务域统一管理。
  - 画像研报数量和运营流程足够独立。
- 不选原因：当前需求围绕竞品变化和单个竞品生成，留在 `/intelligence` 的上下文里更直接。

## 4. 决策依据

- `requirements/raw.md / 原始需求`：要求上传/更新用户画像、选择竞品手动生成研报、研报组件库更好看、排版更美观。
- `requirements/raw.md / 澄清记录 / 研报触发范围`：手动为主，变化后提示可生成，不自动生成与通知。
- `requirements/raw.md / 澄清记录 / 用户画像录入方式`：结构化表单 + Markdown 补充说明，保存后展示当前生效状态和更新时间。
- `requirements/raw.md / 澄清记录 / 研报组件库选型口径`：选择 Markdown 报告组件库。
- `.aisdlc/project/components/web-client.md`：`nova-web` 使用 Vue 3、Vite、Element Plus、Pinia、UnoCSS，`/intelligence` 是静态路由。
- `.aisdlc/project/products/competitive-intelligence-workbench.md`：竞品情报工作台覆盖竞品资料、监控目标、采集任务、变化报告、反馈与知识写回状态。
- `package.json`：当前已有 `element-plus`、`vue-element-plus-x`、`@vueuse/core`、`echarts`，尚无独立 Markdown 报告阅读组件依赖。

## 5. 验证清单

- V-FE-001 Markdown 报告组件兼容性
  - 风险/假设：新增 Markdown 组件可能与当前 Vue 3、Vite 6、TypeScript 和样式体系不兼容。
  - 方法：在 I1 计划中确定候选组件，并在实现前用最小页面验证渲染、样式、按需引入和 build。
  - 成功/失败信号：`pnpm build` 通过，报告正文支持标题、目录或锚点、表格、引用、代码块为成功；构建失败或样式冲突严重为失败。
  - Owner：DEV-FE
  - 截止：I1 计划评审后 2 天
  - 触发动作：失败则切换到现有 Markdown 样式 + 自研轻量渲染容器。

- V-FE-002 研报详情信息层级可读
  - 风险/假设：仅用 Markdown 正文可能无法突出摘要、证据、画像影响和建议动作。
  - 方法：实现详情页后用一份含完整结构化字段的样例报告做桌面与移动视口走查。
  - 成功/失败信号：用户无需滚动全文即可看到摘要、画像版本、竞品、生成状态和关键结论为成功；关键信息淹没在长文中为失败。
  - Owner：DEV-FE
  - 截止：I2 首轮 UI 完成当天
  - 触发动作：失败则把摘要、证据、建议动作提升为独立区块，Markdown 仅承载正文。

- V-FE-003 既有工作台核心路径不回归
  - 风险/假设：在 `/intelligence` 大页面增量扩展可能影响竞品列表、详情抽屉、目标采集、任务记录和普通报告路径。
  - 方法：执行 `pnpm build`，并按竞品情报 smoke 路径打开竞品详情、查看目标、采集历史和普通报告动作。
  - 成功/失败信号：既有路径可操作且无控制台阻断错误为成功；任一既有入口不可用为失败。
  - Owner：DEV-FE
  - 截止：I2 完成当天
  - 触发动作：失败则拆分画像研报状态，避免污染现有 selected competitor / report state。

- V-FE-004 画像表单录入负担可接受
  - 风险/假设：轻量画像字段若排布不佳，用户仍会觉得表单太重。
  - 方法：用空画像、部分字段、完整画像三种状态走查编辑保存体验。
  - 成功/失败信号：用户能在一个视图内完成核心字段和 Markdown 补充说明编辑为成功；需要频繁切换或字段含义不清为失败。
  - Owner：PM + DEV-FE
  - 截止：R2/原型评审或 I2 UI 首轮完成后 1 天
  - 触发动作：失败则重组字段分组或收敛为更少必填项。

## 7. Impact Analysis

### 7.1 受影响模块

| 模块 | 影响类型 | 关键不变量 | stale? |
|------|----------|-----------|--------|
| `web-client` | 新增页面区块、API 类型、Markdown 报告组件依赖 | `pnpm build` 先 `vue-tsc -b` 再 `vite build`；`/intelligence` 是静态路由；DTO 是前端契约入口 | no |
| `competitive-intelligence` | 新增画像管理和画像研报消费入口 | 前端通过 `src/api/intelligence` 调用 `/intelligence/**`；API unwrap 兼容 `data` 和 `rows` | no |
| `chat-agui` | 不作为主路径 | 不把画像研报生成绑定到聊天/AgentScope UI | no |

### 7.2 需遵守的不变量

- `nova-web` 使用 Vue 3、Vite、Element Plus、Pinia 与 UnoCSS，新增组件库必须兼容现有构建和样式体系（来源：`.aisdlc/project/components/web-client.md#tl-dr`）。
- `/intelligence` 静态路由挂载竞品情报页，新功能应在该工作台内增量扩展（来源：`.aisdlc/project/components/web-client.md#api-contract`）。
- 前端 DTO 枚举和 API wrapper 是契约入口，新增画像和画像研报类型应落在 `src/api/intelligence`（来源：`.aisdlc/project/components/web-client.md#data-contract`）。
- 竞品情报 API unwrap 兼容 `data` 和 `rows`，新增接口不应破坏既有 wrapper 习惯。
- 切换竞品时不得泄漏上一个竞品的 targets/tasks/reports；画像研报状态同样需要按 competitor 上下文隔离。

### 7.3 跨模块影响

- 后端需要提供当前画像、保存画像、生成画像研报、研报历史、研报详情接口；前端 API 类型与字段命名需对齐。
- Markdown 报告组件要求后端提供 Markdown 正文，同时结构化字段用于摘要、证据、画像影响和建议动作区块。
- 画像版本策略要求前端在当前画像和研报详情中展示版本号、版本时间或等价信息。
- 新增依赖会影响 `package.json`、lockfile 和构建体积，需要在实现阶段验证。

### 7.4 Context Gaps

- None. 必读项目知识库文件已读取；具体 Markdown 组件包名进入 `V-FE-001` 验证，不作为方案阻塞。

## 8. Mini-PRD

- **MVP 范围 In**：
  - 在竞品情报工作台新增我方用户画像入口。
  - 用户画像编辑包含轻量结构化字段：目标用户、核心场景、关键痛点、购买/使用动机、决策因素、反对理由、产品定位、重点关注指标。
  - 用户画像编辑包含 Markdown 补充说明。
  - 保存后展示当前生效状态、更新时间和版本信息。
  - 用户选择单个竞品手动生成画像研报。
  - 竞品变化后可在相关上下文提示“可生成画像研报”。
  - 展示研报生成中、成功、失败状态。
  - 展示画像研报历史和在线详情。
  - 研报详情使用 Markdown 报告组件，并展示摘要、关键洞察、用户画像影响、证据卡片、建议动作和生成元信息。
- **MVP 范围 Out**：
  - 竞品变化后自动生成画像研报。
  - 文件上传与自动解析。
  - 画像版本回滚、差异对比和恢复旧版。
  - PDF/Word/打印优化/分享链接。
  - 聊天/AgentScope 作为画像研报首版主入口。
- **验收标准（AC）**：
  - AC-UPR-FE-001：用户能进入我方用户画像入口并编辑结构化字段和 Markdown 补充说明。
  - AC-UPR-FE-002：用户画像保存后能看到当前生效状态、更新时间和版本信息。
  - AC-UPR-FE-003：用户能选择单个竞品并手动触发画像研报生成。
  - AC-UPR-FE-004：画像研报生成过程有明确 loading、success、error 状态。
  - AC-UPR-FE-005：画像研报历史可查看，并能打开详情。
  - AC-UPR-FE-006：研报详情排版美观，包含摘要、关键洞察、用户画像影响、证据、建议动作和生成元信息。
  - AC-UPR-FE-007：首版不出现导出、分享、打印优化、回滚、文件上传入口。
  - AC-UPR-FE-008：新功能不破坏既有竞品列表、竞品详情、监控目标、采集历史、任务记录和普通报告查看路径。
- **交互变化结论**：存在新增交互，主要集中在 `/intelligence` 内的画像编辑、竞品手动生成和研报详情阅读；建议后续补 PRD 或原型以消除布局与状态歧义。
- **影响面**：`src/pages/intelligence/index.vue`、`src/api/intelligence/index.ts`、`src/api/intelligence/types.ts`、`package.json`、Markdown 报告组件样式与构建配置。
